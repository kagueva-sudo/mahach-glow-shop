## Доработки админ-панели

### 1. Заявки — поиск и фильтры
В `src/routes/admin.tsx` над списком добавить:
- Текстовый поиск по имени и телефону (фильтрация на клиенте — данные уже загружены).
- Селект статуса: «Все / Новые / В работе / Готовы / Отменены».
- Фильтр по периоду: «Все / Сегодня / Неделя / Месяц».
- В табе «Заявки» — счётчик новых: `Заявки (3 новых)`.

### 2. Товары — загрузка фото с компьютера
В компоненте `ProductEditor`:
- Рядом с полем «URL изображения» добавить кнопку **«Загрузить с компьютера»**.
- Использовать тот же bucket `site-assets` (он уже есть и приватный, доступен только админам).
- Файл → путь `products/{timestamp}-{rand}.{ext}` → создание долгого signed URL (100 лет, как для hero) → автоматически записывается в поле `image_url`.
- Превью загруженного изображения под полем.
- Валидация: только image/*, до 8 МБ.

### 3. Профиль администратора
Новый таб **«Профиль»** в админке:
- Текущий email (read-only из `supabase.auth.getUser()`).
- Форма **«Сменить пароль»**: поля «Новый пароль» + «Подтверждение», вызов `supabase.auth.updateUser({ password })`. Валидация: минимум 8 символов, совпадение.
- Форма **«Сменить email»**: новый email → `supabase.auth.updateUser({ email })`. Предупреждение, что на новый адрес придёт письмо для подтверждения.

### Технические детали

- Все изменения — только в `src/routes/admin.tsx` (UI) и при необходимости небольшая утилита загрузки в Storage прямо в компоненте (как уже сделано для hero).
- BD-миграции не нужны: bucket `site-assets` существует, RLS-политики уже ограничивают доступ админами, `auth.users` управляется Supabase.
- Серверные функции трогать не требуется — Supabase Auth API вызывается напрямую с клиента (`supabase.auth.updateUser`).
- Поиск/фильтры — клиентская фильтрация над уже загруженным массивом заявок (объём небольшой).

### Что НЕ входит в эту итерацию
- Приглашение других администраторов (по решению пользователя).
- Экспорт заявок в CSV.
- Drag-and-drop сортировка товаров (есть поле `sort_order` в редакторе).
